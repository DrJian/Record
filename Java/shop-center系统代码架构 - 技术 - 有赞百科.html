    shop-center系统代码架构

    转至元数据结尾
    • 创建： qiandaoyin，最新修改于： 三月 21, 2017
    转至元数据起始

    shop-center系统代码架构

    遵循原则：

    1. 职责边界清晰；
    2. 功能内聚；
    3. 提高代码复用率；
    4. 领域模型驱动设计（模型抽象）；
    5. 便于后期代码维护。

     

    各层的职责阐述：

    广义controller层
    依赖 biz-service，两者职责界定，biz-service提供具体专一业务功能，controller是为外部提供一个具体的服务，具体职责如下：
    1. 入口参数基础校验；
    2. 系统异常捕获处理；
    3. 对外业务模型转换（领域模型-》业务模型）；
    4. 业务功能编排（可能调用不限于一个biz-service服务）；
    模块类的命名：
    xxx-web: XxxContronller;
    xxx-biz-impl:实际为Xxx-api接口定义的实现, XxxServiceImpl;
    xxx-message:监听外部消息事件逻辑处理，XxxMessageHandler；
        
    Biz-Service层
    依赖Core-service，这两层职责界定在什么地方，core-Service主要职责在于领域模型提供的服务，biz-service提供具体专一业务功能，那领域服务和业务功能区分在什么：
    理论我这里就不扯了，这里举个例子吧
    创建交易的时候有两个模块是然不过去的，订单模型 和 库存模型，订单模型会提供订单创建、查询、更新功能；库存模型会提供扣减、添加、查询功能。那么交易创建时候biz-service调用前面两个模型，而那两个模型就是定义在core-Service.
    模块类的命名：
    biz-Xxx: XxxInnerService，这里加“Inner”为了和上面“biz-impl”对外服务做出区分，表明这是内部服务；
       
    Core-Service层
    依赖于Common，Core-Service主要职责在于领域模型提供的服务，common只提供一些基层简单的服务，例如DB的增删改查，外部服务调用等。
    例如：继续上面例子，biz -> core层创建订单，领域服务首先将外部订单模型转换成数据库需要的数据模型，然后调用comm-dal保存到DB中，如果这里有异常，领域服务需要自己消化这些逻辑，然后返回异常以及相应的错误码。
    模块类的命名：
    Core-Service:XxxDomainService，这里加“Domain”为了和其他服务区分，表明是领域服务，输出的对象一般为XxxDTO；
      
    Common层
    common-util：工具类：日志，配置管理，缓存工具，httpClient工具等等
    common-client：外部服务引用，依赖于common-util（http服务依赖于httpClient）输出服务命名：XxxServiceClient；
    common-dal：数据访问层（包含缓存数据），依赖于common-util（缓存工具）;定义XxxPO和XxxDAO，这里强调DAO的方法命名，selectXxx；updateXxx；deleteXxx；insertXxx。
    Xxx-api(Xxx-idl)：外部接口XxxService，对外输出对象信息如：XxxBO，XxxResult；
      
    注：
    *  PO：(Persistent Object)用于持久层
    *  DTO：(Data Transfer Object)这里是领域模型对象,这里没有用DO(Domian Object),担心大家和Data Object缩写混淆
    *  BO: (Business Object)用于api层输出业务对象
